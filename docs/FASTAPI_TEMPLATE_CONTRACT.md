# FastAPI Template Contract

This document defines the contract reviewers should expect from every
service generated by this template. It is intentionally short: the source
of truth is still the scaffolded code and its tests, but this page makes
the serving expectations explicit for adopters, reviewers, and agents.

## Scope

The scaffolded service ships a FastAPI serving layer at repo root:

- `app/main.py` owns the FastAPI app, lifespan, health/readiness, model
  management endpoints, CORS, error envelope, and tracing middleware.
- `app/fastapi_app.py` owns the inference router, metrics, model artifact
  loading, feature parity, SHAP wrapper, and prediction logging.
- `app/schemas.py` owns public request/response models.
- `src/<service>/training/features.py` owns the reusable
  `FeatureEngineer` used by both training and inference.

Do not add a parallel API template. Improve this contract instead.

## Required endpoints

Every scaffolded service must expose:

| Endpoint | Purpose | Contract |
|----------|---------|----------|
| `POST /predict` | Single inference | Uses `ThreadPoolExecutor`, validates input twice, returns `prediction_id`, score, risk level, and model version. |
| `POST /predict_batch` | Batch inference | Uses one executor call for the batch, validates the full batch atomically, logs each prediction. |
| `GET /health` | Liveness | Always 200 while the process is alive; never doubles as readiness. |
| `GET /ready` | Readiness | 200 only after model artifacts are loaded and warm-up completed; otherwise 503. |
| `GET /metrics` | Prometheus scrape | Public inside the cluster; restricted by NetworkPolicy rather than API auth. |
| `GET /model/info` | Model metadata | Protected by API auth when `API_AUTH_ENABLED=true`. |
| `POST /model/reload` | Admin reload | Hidden unless admin API is explicitly enabled and authorized. |

## Non-negotiable invariants

- D-01/D-03: Kubernetes runs a single Uvicorn worker; CPU-bound model
  calls run through `asyncio.run_in_executor()` and
  `ThreadPoolExecutor`.
- D-23: `/health` and `/ready` are split. Readiness gates on both
  model load and warm-up.
- D-24: SHAP `KernelExplainer` is built once at startup when background
  data exists; explanations stay opt-in through `?explain=true`.
- D-21/D-22: prediction logging is fire-and-forget; logging failures
  never break the HTTP response.
- Feature parity: inference calls the same `FeatureEngineer` contract as
  training before `predict_proba`. Raw features are allowed only through
  a documented serialized-pipeline exception with
  `FEATURE_ENGINEERING_REQUIRED=false`.
- Modelless startup is a CI/dev convenience only.
  `ALLOW_MODELLESS_STARTUP=true` must fail in staging, production, or
  any non-dev environment.
- CORS is default-deny. `CORS_ORIGINS` must be explicit.
- Non-2xx responses use the stable error envelope unless an adopter is
  running a documented migration.

## Customization checklist

After scaffolding, customize in this order:

1. Update `app/schemas.py` with public API fields, examples, and
   response semantics.
2. Update `src/<service>/schemas.py` with the Pandera data contract and
   observed ranges from EDA.
3. Implement `src/<service>/training/features.py` so
   `transform()` and `transform_inference()` produce the same columns.
4. Implement `src/<service>/training/model.py` and `train.py`, then run
   quality gates before producing `model.joblib`.
5. Update tests for the real schema, including at least one valid
   `/predict` payload and one bad payload that must return 422.
6. Configure auth, prediction logging backend, model path, background
   data path, and metric prefix before cluster deployment.

## Reviewer evidence

A staff/lead reviewer should see all of the following before accepting a
service PR:

- `pytest tests/test_fastapi_template_contract.py -v` passes after
  scaffolding.
- `pytest tests/test_api.py tests/test_auth.py tests/test_input_validation.py -v`
  passes with the real service schema.
- `pytest tests/test_training.py -k inference_uses_same_features -v`
  proves train/inference feature parity.
- `SCAFFOLD_SMOKE=1 bash scripts/test_scaffold.sh` passes in the
  template repo.
- Cluster smoke validates `/health`, `/ready`, `/predict`, and
  `/metrics` with the real auth posture and a schema-valid payload.

This evidence supports the README's L1-L3 "Designed-ready" claim. It
does not claim L4 production rollout; real GKE/EKS execution remains
adopter-owned evidence.
