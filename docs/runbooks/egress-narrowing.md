# Runbook — NetworkPolicy Egress Narrowing (per cloud, per region)

- **Authority**: ADR-020 §R5-M3 follow-up; VALIDATION_LOG Entry 005.
- **Mode**: CONSULT (edits cluster-level network policy; changes can
  break pod egress to cloud APIs if CIDRs are wrong).
- **Scope**: replace the residual CIDR allowlists shipped by the
  template overlays (`templates/k8s/overlays/{gcp,aws}-{staging,prod}/patch-networkpolicy.yaml`)
  with the exact per-region prefix lists from each cloud's public
  IP-ranges feed, so the service's egress is tightened to the
  minimum set of endpoints the init container + sidecar actually need.
- **Approver**: Platform Lead + Security Engineer.
- **Audit trail**: each successful execution writes an entry to
  `VALIDATION_LOG.md` and emits an `AuditEntry` via
  `scripts/audit_record.py`.

---

## Why this runbook exists

R5-M3 shipped per-non-dev-overlay NetworkPolicy patches that narrow
the base's permissive `0.0.0.0/0:443` cloud-storage egress to
cloud-specific CIDR residuals:

- GCP overlays pin `199.36.153.4/30` + `199.36.153.8/30` (Private
  Google Access VIPs) + `34.0.0.0/8` residual.
- AWS overlays pin `52.0.0.0/8` + `54.0.0.0/8 except 54.64.0.0/11`.

Those residuals are **better than `0.0.0.0/0`** (they exclude all
non-googleapis / non-AWS IPs) but **broader than necessary**. Real
production deploys in regulated environments (fintech, healthcare,
government) must tighten further to the exact CIDRs used by:

- the region's Artifact Registry / ECR endpoint;
- the region's Cloud Storage / S3 endpoint;
- MLflow tracking (if off-cluster);
- Secret Manager / Secrets Manager.

This runbook is the operator procedure for doing that. The template
cannot ship the tightened CIDRs because they are per-region, per-cloud,
and change over time — AWS publishes a signed `ip-ranges.json` feed
that must be polled and re-applied; GCP's allocations shift with
regional expansion.

---

## Pre-conditions

- `kubectl` authenticated against the target cluster.
- `jq` available locally for AWS IP-ranges parsing.
- Read access to `ip-ranges.amazonaws.com/ip-ranges.json` (public).
- A representative test request the service issues during startup
  (e.g. an init-container `aws s3 cp` or `gcloud storage cp`) that
  will FAIL closed if the policy is too tight.

## Procedure A — AWS (EKS) per-region CIDR pinning

```bash
REGION="${AWS_REGION:-us-east-1}"
OVERLAY_DIR="templates/k8s/overlays/aws-prod"   # or aws-staging

# 1. Pull the authoritative IP-ranges feed.
curl -fsSL -o /tmp/aws-ip-ranges.json https://ip-ranges.amazonaws.com/ip-ranges.json

# 2. Extract S3 + EC2 + STS prefixes for the target region.
jq -r --arg r "$REGION" '
  .prefixes[]
  | select(.region == $r and (.service == "S3" or .service == "AMAZON" or .service == "EC2"))
  | .ip_prefix
' /tmp/aws-ip-ranges.json | sort -u > /tmp/aws-cidrs.txt

wc -l /tmp/aws-cidrs.txt   # typical: 40-120 CIDRs per region
```

- `AMAZON` covers Secrets Manager, STS, and most regional control-plane endpoints.
- `S3` is the per-region S3 data plane.
- `EC2` is used by ECR Docker-registry endpoints.

### Regenerate the overlay patch

Convert the CIDR list into a JSON 6902 `replace` patch that swaps the
existing `egress[3]` residual:

```bash
python3 - <<'PY'
import json, pathlib
cidrs = pathlib.Path("/tmp/aws-cidrs.txt").read_text().strip().splitlines()
to = [{"ipBlock": {"cidr": c}} for c in cidrs]
patch = [{
    "op": "replace",
    "path": "/spec/egress/3",
    "value": {"to": to, "ports": [{"protocol": "TCP", "port": 443}]},
}]
pathlib.Path("/tmp/patch-networkpolicy.yaml").write_text(
    "# Auto-generated by docs/runbooks/egress-narrowing.md on $(date -u).\n"
    "# Source: https://ip-ranges.amazonaws.com/ip-ranges.json\n"
    "# Region: $AWS_REGION — regenerate quarterly or when AWS publishes\n"
    "# a new ip-ranges.json release (follow the 'createDate' field).\n"
    + "---\n" + __import__("yaml").safe_dump(patch, sort_keys=False),
    encoding="utf-8",
)
print(f"Wrote patch with {len(cidrs)} CIDRs")
PY
```

Copy `/tmp/patch-networkpolicy.yaml` over
`templates/k8s/overlays/<overlay>/patch-networkpolicy.yaml` in a
feature branch, open a PR, and run `kustomize build` through the
normal PR pipeline — `test_networkpolicy_egress_hygiene.py` will
confirm the new patch does not contain `0.0.0.0/0`.

### Validation

```bash
# After deploy:
kubectl --context=<cluster> -n <namespace> get networkpolicy \
  <service-name>-network-policy -o yaml | grep -c 'cidr:'
# Expect: the number you wrote + 3 (kube-dns + MLflow + pushgateway).
```

Force a known-good egress target via the init container and assert it
succeeds:

```bash
kubectl -n <namespace> exec -it deploy/<service-name> -c init-container -- \
  aws s3 ls s3://<your-model-bucket>/ --region "$REGION"
```

Force a known-bad target to confirm the policy blocks it:

```bash
kubectl -n <namespace> run denier --image=curlimages/curl:8.10.1 \
  --restart=Never --rm -it -- curl -sSL --max-time 5 https://google.com
# Expect: timeout / connection refused (egress to non-AWS blocked).
```

## Procedure B — GCP (GKE) per-region CIDR pinning

GCP does not publish a single `ip-ranges.json` equivalent. The two
canonical paths:

### B.1 — Private Google Access (preferred)

If the GKE cluster uses Private Google Access, the googleapis endpoints
resolve to fixed VIPs:

- `199.36.153.4/30` — `restricted.googleapis.com` (VPC-SC compatible)
- `199.36.153.8/30` — `private.googleapis.com` (standard)

These are already in the shipped overlay patch. For a Private-Google-
Access-only cluster, **no tightening is needed** — the current
residual `34.0.0.0/8` can be removed entirely:

```yaml
# templates/k8s/overlays/gcp-prod/patch-networkpolicy.yaml (tightened)
- op: replace
  path: /spec/egress/3
  value:
    to:
      - ipBlock:
          cidr: 199.36.153.4/30
      - ipBlock:
          cidr: 199.36.153.8/30
    ports:
      - protocol: TCP
        port: 443
```

### B.2 — Dedicated egress NAT CIDR

If the cluster egresses through a Cloud NAT with a reserved external
IP range, pin egress to that range:

```bash
# Resolve the NAT's external IPs.
gcloud compute routers get-nat-mapping-info <router-name> \
  --region=<region> --format=json \
  | jq -r '.result[].natIpPortMappings[].natIp' | sort -u
```

Then substitute each returned IP (as a `/32` or grouped `/28`) into
the overlay patch `to:` block.

### B.3 — Layer-7 FQDN policy (most hardened)

For GKE Dataplane V2 (Cilium-based) clusters, layer a CiliumNetworkPolicy
with `toFQDNs` on top of the base NetworkPolicy:

```yaml
apiVersion: cilium.io/v2
kind: CiliumNetworkPolicy
metadata:
  name: "{service-name}-egress-fqdn"
spec:
  endpointSelector:
    matchLabels:
      app: "{service-name}"
  egress:
    - toFQDNs:
        - matchName: "storage.googleapis.com"
        - matchPattern: "*.googleapis.com"
        - matchName: "{REGION}-docker.pkg.dev"
      toPorts:
        - ports:
            - port: "443"
              protocol: TCP
```

This is the only mechanism in Kubernetes that expresses true FQDN
egress control; the vanilla NetworkPolicy's `ipBlock` is CIDR-only.

## Update cadence

- **AWS**: re-run Procedure A once per quarter, or within 7 days of
  AWS publishing a new `createDate` in `ip-ranges.json`. Schedule
  via a calendar reminder or a nightly CI job that diffs the feed.
- **GCP Private Google Access**: no cadence needed — the VIPs are
  fixed contractually.
- **GCP dedicated NAT**: re-run whenever the NAT external IPs change
  (rare; only on NAT re-creation or IP reservation changes).

---

## When NOT to tighten

Keep the residual shipped by the template (do NOT apply this runbook)
when the environment is:

- a dev cluster used for local iteration;
- a smoke-test ephemeral cluster (kind, k3d, minikube);
- a cluster whose only outbound traffic is already mediated by an
  HTTP forward proxy with its own allowlist.

In those cases the residual is already below the real exposure floor
and tightening adds operational cost with no security benefit.

## Audit entry

After applying a tightened patch and confirming validation above, record:

```bash
python3 scripts/audit_record.py \
  --agent operator \
  --operation "egress-narrowing" \
  --environment "<staging|prod>" \
  --base-mode CONSULT \
  --final-mode CONSULT \
  --result success \
  --inputs '{"cloud":"aws","region":"us-east-1","source":"ip-ranges.json"}' \
  --outputs '{"overlay":"aws-prod","cidrs_count":72,"patch_sha256":"<sha>"}' \
  --approver "<lead-github-handle>"
```

And append to `VALIDATION_LOG.md` with the overlay path, the `kubectl get`
CIDR count, and the validation test output.
