# MCP Portability

**Authority**: `docs/decisions/ADR-023-agentic-portability-and-context.md`
**Sources**:
- `templates/config/mcp_registry.yaml` — capability-centric MCP list
- `templates/config/surface_capabilities.yaml` — surface × capability matrix
- `scripts/mcp_doctor.py` — diagnostic + doc renderer

---

## Why this layer exists

Before F4, MCP setup was documented inside `AGENTS.md §MCP Integrations`
and nowhere else. Adding a fourth agent surface (Codex) or a fifth
(Antigravity) would have meant another ad-hoc paraphrase of that table
in a fourth `.md` file, with drift compounding as version numbers
shift. The registry flips the arrow: MCPs are declared once in YAML;
every surface documents how to consume that canonical declaration.

## Invariants

- **I-3 (ADR-023 §3)** — MCP install is **never automatic**.
  `scripts/mcp_doctor.py` validates configuration and renders docs;
  it **never invokes an installer**, writes to a credential store,
  or configures access to production.
- **D-17 / D-18** — credentials live in Secret Manager, never in the
  registry, never in the checked-in example configs.
- **AUTO-class entry points only** — every diagnostic (`check`,
  `doctor`, `render-docs`) is read-only and re-runnable.

## Three commands

```bash
make mcp-check       # fast pass/fail (exit 0 / 1) for CI
make mcp-doctor      # long-form human-readable report
make mcp-render-docs # regenerate the generated section below
```

Underneath, all three call `scripts/mcp_doctor.py`. The script never
reaches the network, never spawns a subprocess, never writes outside
`docs/agentic/mcp-portability.md` (and only that file, only when
explicitly requested with `--mode render-docs`).

## Cross-checks enforced

1. Every `required_for` / `recommended_for` entry references a skill
   or workflow id that exists in `agentic_manifest.yaml`.
2. Every surface in `mcp_registry.yaml` also appears in
   `surface_capabilities.yaml` (and vice versa).
3. Every MCP's `install_mode` covers every known surface.
4. For every skill: every surface that claims the skill supports every
   capability the skill requires (from
   `surface_capabilities.skill_capability_requirements`).

Failure of any cross-check exits `1` and prints the offending
`mcps.<id>` or `skills.<id>` anchor for line-accurate debugging.

## Install-mode semantics

| Mode | Meaning | Example |
|------|---------|---------|
| `documented` | Surface's docs describe setup; adopter runs it manually. | Windsurf reads `~/.codeium/windsurf/mcp_config.json`. |
| `user_confirmed` | Surface supports a per-request confirmation dialog before invoking the MCP. Agent may NOT skip. | Codex's MCP bridge prompts before a `kubectl` call. |
| `optional` | Surface works without the MCP; feature degrades (e.g. no browser-smoke in `release-checklist`). | Playwright on Codex. |
| `planned` | Surface does not support this MCP yet; slot reserved. | Antigravity, when added. |

Three modes **deliberately absent**: `automatic`, `silent`, and
`credentialed`. Any PR reintroducing those terms must bump ADR-023 I-3.

## How to add a new MCP (contributor workflow)

1. Add the entry to `templates/config/mcp_registry.yaml` with:
   - `purpose` (one line, declarative)
   - `risk_mode` (`AUTO` / `CONSULT` / `STOP`)
   - `authority` (anchor into `AGENTS.md#MCP Integrations` or an ADR)
   - `required_for` / `recommended_for` (manifest skill/workflow ids)
   - `install_mode` entries for **every** known surface
2. Run `make mcp-check` — must exit 0.
3. Run `make mcp-render-docs` — rewrites the generated section of
   this file. Commit the result in the same PR.
4. Update `AGENTS.md §MCP Integrations` if the MCP is globally
   relevant. Optional when the MCP is niche (e.g. playwright).
5. Adopters consume the change by re-reading this doc and manually
   updating their surface's MCP config. **The agent never does this
   for them** (I-3).

## How to add a new surface

1. Add the surface to `surface_capabilities.yaml` with:
   - `supports:` list of capabilities it can execute
   - `config_targets:` paths where the surface reads its config
2. Add the surface to every MCP's `install_mode` in `mcp_registry.yaml`.
3. Update `agentic_manifest.yaml` surfaces entry (status + pointer).
4. Add a `.<surface>_context.md` pointer file (<= 60 lines,
   references `AGENT_CONTEXT.md` per F3 format validator).
5. `make mcp-check` + `python3 scripts/validate_agentic_manifest.py --strict`
   must both exit 0 before the PR merges.

# MCP Portability Matrix

Source: `templates/config/mcp_registry.yaml` + `templates/config/surface_capabilities.yaml`.

## Registered MCPs

| MCP | Purpose | Risk | Required for |
|-----|---------|------|--------------|
| `github` | Read PRs, CI logs, issues, release evidence, status checks | `AUTO` | `release-checklist`, `security-audit`, `rule-audit` |
| `kubectl` | Run read-only kubectl get/describe/logs against the cluster | `CONSULT` | `debug-ml-inference`, `incident`, `rollback` |
| `terraform` | Registry lookup, provider/module metadata, plan validation | `AUTO` | `release-checklist`, `cost-audit` |
| `prometheus` | Query live metrics for dynamic risk context + drift analysis | `AUTO` | `drift-detection`, `performance-degradation-rca`, `incident`, `performance-review` |
| `playwright` | End-to-end browser smoke-tests + scripted login flows | `AUTO` | — |
